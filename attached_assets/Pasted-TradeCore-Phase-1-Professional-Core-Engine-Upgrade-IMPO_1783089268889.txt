TradeCore Phase 1 – Professional Core Engine Upgrade

IMPORTANT: Upgrade the EXISTING trading engine in-place. Do NOT create a second engine or duplicate strategy logic. Reuse the current architecture wherever possible while replacing weak logic with more advanced implementations.

Do NOT redesign the UI.
Do NOT deploy anything to the VPS.
Do NOT remove existing functionality.
Do NOT modify my .env, Binance API keys, database credentials, or production configuration.
Keep paper trading, live trading, REST API, dashboard, database, and existing architecture fully compatible.

Objective:
Transform the current trading engine into a professional quantitative trading engine with smarter decision-making, adaptive risk management, and multi-timeframe confirmation while preserving the existing TradeCore architecture.

Requirements:

1. Multi-Timeframe Analysis
Replace the current single-timeframe strategy with multi-timeframe confirmation.

Use:
- 1m for entries
- 3m for momentum
- 5m for trend
- 15m for confirmation
- 1h for macro trend

Only enter trades when lower timeframe signals agree with higher timeframe direction.

2. Weighted Indicator Voting System
Replace the current RSI-heavy logic with a weighted voting engine.

Each indicator returns:
- Bullish
- Bearish
- Neutral

Indicators:
- EMA 20
- EMA 50
- EMA slope
- RSI
- MACD Histogram
- ADX
- ATR
- VWAP
- Bollinger Bands
- Volume Spike
- Candle Structure
- Support/Resistance
- Trend Alignment

Each indicator contributes a configurable confidence weight.

Generate one final confidence score between 0–100.

No trade below a configurable confidence threshold.

3. Dynamic Confidence
Remove hardcoded confidence values (70, 80, 90, 100).

Calculate confidence mathematically from indicator agreement.

Every trade should have its own unique confidence score.

4. Market Regime Detection
Before searching for entries determine market condition.

Supported regimes:
- Strong Trend
- Weak Trend
- Range
- High Volatility
- Low Volatility

Trend strategies should only run during trends.

Range strategies should only run during ranging markets.

Skip poor market conditions.

5. ATR-Based Risk Management
Replace static SL/TP with adaptive ATR calculations.

Implement:
- Stop Loss = ATR × configurable multiplier
- Take Profit = ATR × configurable multiplier
- Optional trailing stop
- Adaptive volatility-based exits

6. Professional Position Sizing
Position size must consider:
- Risk %
- Account balance
- ATR stop distance
- Binance minimum order size
- Maximum simultaneous exposure
- Maximum portfolio risk

7. Intelligent Coin Ranking
Scan every supported symbol.

Score every opportunity.

Rank opportunities by confidence.

Trade only the highest-quality opportunities.

Maximum concurrent positions should remain configurable.

8. Trade Cooldown
Implement:
- Symbol cooldown after exit
- Ignore duplicate entries
- Prevent immediate re-entry
- Prevent overtrading

9. Professional Logging
Rejected trades should log:
- Confidence
- Indicator votes
- Market regime
- Trend
- ATR
- Exact rejection reason

Accepted trades should log:
- Confidence
- Indicator breakdown
- Entry reason
- Stop Loss
- Take Profit
- Position size
- Risk %

10. Backtesting Compatibility
The backtesting engine MUST use the exact same strategy logic as live trading.

Do NOT duplicate strategy code.

Live trading and backtesting must always produce identical decisions given identical market data.

11. Code Quality
Refactor duplicated calculations into reusable modules.

Remove duplicated code.

Remove magic numbers.

Move configurable values into configuration.

Maintain strong TypeScript typing.

Keep the architecture modular and maintainable.

12. Validation
After implementation:

- Rebuild the API server
- Rebuild the frontend
- Verify successful compilation
- Run a one-month backtest

Provide a report showing:
- Total trades
- Win rate
- Net profit
- Profit factor
- Maximum drawdown
- Expectancy
- Average trade duration
- Average confidence
- Top-performing symbols
- Worst-performing symbols

Do NOT deploy to the VPS.

Only update the Replit project.

The objective is to evolve the existing TradeCore engine into a professional quantitative trading engine without breaking compatibility or creating duplicate systems.
