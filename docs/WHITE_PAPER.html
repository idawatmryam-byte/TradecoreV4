01

Executive summary

TradeCore Pro is an autonomous, multi-strategy trading platform written in TypeScript as a pnpm monorepo. A single codebase runs two fully independent markets — a crypto section on Binance (spot and USDⓈ-M futures, long and short) and a forex section on OANDA (majors, gold, and USD-quoted indices) — through one shared decision core. Each market has its own connection, strategy tuning, positions, trade log, decision journal, and on/off switch; the two can be configured side by side.

The platform is built around a professional's discipline rather than a black box. The operator states risk in dollars — the most they will lose and aim to win per trade — and the engine derives position size, stop price, and target from that plan and the instrument's structure. Every trade the engine considers, whether taken or declined, is logged with its reasoning; every closed trade is automatically analysed and graded; and the same code that trades live also drives the backtester, so simulated results reflect real behaviour.

02

System overview & the two-axis model

The whole system is organised around two orthogonal concepts. Keeping them separate is what lets one codebase serve two very different markets without special-casing.

  • Section (crypto | forex) scopes everything a user owns — configuration, strategies, positions, trade history, decisions, statistics, and the engine's on/off state. It is carried on every API request via an X-Section header and threaded through every database query.
  • Broker × market type (binance/oanda with spot/futures/forex) selects the execution adapter and sizing semantics — which exchange to call, how orders are shaped, and whether leverage and liquidation apply.

Because a market type of forex is treated by the sizing math like un-leveraged spot, the existing dollar-risk engine applies unchanged to FX — the only place home currency and market-hours logic enter is at clearly-bounded adapter and gating layers. Crypto behaviour is regression-locked throughout: a deterministic parity harness proves that adding forex changed the crypto engine's output by exactly zero.

03

Architecture & repository structure

The monorepo separates the trading server, the dashboard, the database schema, and a generated API contract. A single Node process serves both the REST API and the built single-page app, so production runs one service.

tradecore-pro
React / Vite dashboard (wouter routing, TanStack Query). Section switcher, cockpit, trade log, decisions feed, analytics, backtesting.
api-server
Express API + the trading engine. One BotEngine per (user, section); ccxt for Binance, a ccxt-shaped adapter for OANDA.
lib/db
Drizzle ORM over PostgreSQL — the single source of truth for users, config, trades, decisions, memory, and backtests.
lib/api-spec
OpenAPI specification → generates a typed React client and Zod validators, so frontend and backend can never silently drift.

The generated contract is the load-bearing idea: an endpoint's request and response shapes are defined once and compiled into both a typed client hook and a runtime validator. A change to the API that the frontend hasn't accounted for fails at compile time, not in production.

3.1The engine registry

Live trading state is held per user and per section. A registry keys each running engine on ${userId}:${section}; a request resolves to its owner's engine and can touch no other. The engine owns the scan loop, ticker polling, open-position tracking, and the exit pipeline for its slice of the world.

04

Data model & multi-tenancy

Every user runs an isolated instance against their own encrypted credentials. Isolation is enforced in the data layer: each user-owned table carries a user_id, and the two markets are further separated by a section column, so no query can cross a tenant or a market boundary by accident.

› users & identities

Accounts, hashed passwords, optional linked OAuth identities, and the read-only demo flag.

› credentials

Binance and OANDA secrets, AES-256-GCM encrypted at rest; only a masked preview is ever returned.

› bot_config

Per-(user, section) settings: risk model, dollar plan, pairs, market type, leverage, desired run state.

› trades & partial exits

Full execution record — planned vs actual SL/TP/qty, fees, slippage, P&L, the exact plan executed.

› strategy_decisions

The journal of every considered trade — executed, approved-not-taken, or rejected with a reason.

› memory & analyses

Hourly stats, a poor-performer blacklist, and an automatic post-trade grade for each closed trade.

› backtest_runs

Backtest results with trades and equity curves, section-scoped, sharing the live-parity engine.

› autopsy_runs

Walk-forward optimisation diagnoses — train/validation windows, verdicts, and evidence.

Account deletion is complete and ordered: the engines stop, then every row the user owns is removed across all tables before the account itself, leaving no orphaned trading state.

05

The decision engine

On each scan the engine turns market data into a decision through an explicit pipeline. Every stage can reject a trade with a stated, recorded reason — the engine is never a black box that simply "didn't trade".

5.1Strategies as reasoning brains

Each strategy is a self-contained decision-maker. Rather than emit a bare signal, a strategy produces a full trade plan: side, entry, a structural stop (placed where the thesis is actually wrong — a broken level, a range midpoint — not an arbitrary percentage), a target, sizing solved from the dollar plan, an expected hold time, and a written rationale. Crypto and forex carry different catalogs, because they are different markets: the crypto book keeps its eight strategies; the forex book keeps the structural strategies that translate and adds a forex-native London-session breakout, while dropping the crypto-microstructure scalpers whose edge depends on 24/7 flow.

5.2The cost-aware quality gate

A single central gate enforces that a trade's reward justifies its risk after round-trip costs. Crucially, the cost model is market-correct: forex spreads and slippage are roughly an order of magnitude tighter than crypto, and the gate uses the active market's real rates. Using the wrong rates here is not cosmetic — charging crypto costs against forex-scale targets rejects every forex trade outright, a failure mode the platform explicitly guards against.

Net reward-to-risk, after round-trip costs // costs widen effective risk and shrink effective reward on both legs
cost  = entry × (2·fee + 2·slippage)
netRR = (grossReward − cost) / (grossRisk + cost)
// a plan below the floor is a structurally poor trade — rejected, and recorded as such
06

Order execution

Execution is where the two markets diverge most, so the platform abstracts the broker behind a common adapter interface and lets each implementation express its venue faithfully.

Binance

Spot and USDⓈ-M futures via ccxt. Because Binance futures has no atomic OCO, protection is placed as independent STOP_MARKET / TAKE_PROFIT_MARKET orders, with a fail-safe that flattens a position if its stop cannot be placed — the engine never holds an unprotected fill.

OANDA

A ccxt-shaped adapter over the v20 REST API. Entry, stop, and take-profit are submitted in a single atomic bracket order, so a fill is protected the instant it exists. Balances in a non-USD home currency are converted at a live rate at one boundary, leaving the engine's internal math entirely in USD.

Broker-native trade identifiers are persisted, not held only in memory, so stop/target replacement and position closes survive a process restart. On startup the engine reconciles its in-memory tracking against the broker's actual open positions, closing the gap that a restart mid-trade would otherwise open.

07

Risk & capital protection

Risk control is layered, and every layer is enforced by the engine rather than left to the operator's discipline.

  • Dollar-denominated sizing. The operator states max loss and target profit per trade; the engine derives quantity, stop price, and target from that plan and the position's notional — one planner shared by the live engine and the backtester.
  • Daily-loss circuit breaker. Once the day's losses hit the configured limit, no new entries open; open positions continue to be managed. State is persisted so a restart cannot silently reset it.
  • Consecutive-violation pause. If realised losses exceed their expected maximum repeatedly, trading pauses and requires a manual reset — a guard against a strategy or market behaving outside its modelled bounds.
  • Futures liquidation safety. Leverage is clamped to the exchange's per-symbol maximum and to the platform's own lower cap; a trade whose stop would sit inside the liquidation distance is refused before it is placed, and margin sufficiency is pre-checked.
  • Market-hours gating (forex). A DST-aware calendar skips closed instruments entirely — no evaluation against a stale Friday close — while still managing any open position, since a resting stop can fill whenever the venue is open.
Design principle

A stopped section's open positions always keep their exchange-side stop-loss and take-profit. They may lose active management (the partial-profit ladder, trailing, time exits) until the section is restarted, but they are never left unprotected.

08

Backtesting, parity & the Optimization Autopsy

The backtester runs the same decision code as live trading — the same signals, the same dollar-risk planner, the same fee and slippage model. Results therefore describe what the live engine would have done, not a parallel simulation that might disagree. Crypto uses public exchange data; forex downloads OANDA candles and models forex costs and market hours.

8.1The Optimization Autopsy

The Autopsy answers a sharper question than a raw backtest: what is wrong with this strategy's current configuration? It sweeps one strategy's real knobs — dollar plan, confidence floor, hold time — on a training window, then re-tests the leading candidates and the current config on a later validation window they were never fitted to. Only a candidate that beats the current configuration out of sample becomes a suggestion.

"No better" is a first-class, honest verdict — the tool is designed to tell the operator when their configuration is already sound, not to manufacture a change. This discipline is deliberately the opposite of an unvalidated auto-tuner that would chase noise.

09

Edge Forensics

A raw win rate is a symptom, not a diagnosis. Edge Forensics decomposes realised performance into named, dollar-quantified causes so the operator improves on evidence.

  • Scratch-adjusted win rate. Break-even exits and washes are separated from real losses — a strategy that mostly scratches can show a frightening raw win rate while its capital is barely moving; the adjusted figure is the truthful one.
  • Leak verdicts. The biggest bleeds are ranked in dollars: fast "noise" stop-outs (stops sitting inside ordinary volatility), fee drag, and negative strategy / symbol / hour cells.
  • Stop-vs-noise audit. Any strategy whose implied stop is tighter than the instrument's current volatility is flagged — a classic cause of pathological win rates.

The output is a to-do list grounded in the account's own history: widen a too-tight stop, retune a bleeding strategy, or stop trading a losing hour.

10

Security

Secrets at rest

Exchange keys are AES-256-GCM encrypted; the server never returns plaintext, only a masked preview. All runtime config comes from a gitignored environment file — nothing sensitive is committed.

Sessions

Stateless, HMAC-signed, HttpOnly session cookies with an expiry. No server-side session store to breach; rotating the signing secret invalidates every outstanding session at once.

Authentication

scrypt password hashing with constant-time verification; login attempts for unknown and known users cost the same, so timing can't enumerate accounts. Login and registration are separately rate-limited.

Supply chain

Dependencies are held to a minimum-release-age policy — a new package version must have been public for a day before it can be installed, blunting the most common npm compromise vector.

Tenant isolation

Every request resolves to exactly one user's engine and data; a demo account is enforced read-only at the server, not merely in the UI, and holds no exchange credentials.

Transport & headers

Security headers, a strict CORS allow-list, and body-size caps guard the single public surface; the trading engine reaches exchanges over authenticated HTTPS only.

11

Testing & verification

Correctness is defended by a suite of deterministic checks that run offline — no exchange required — plus a parity discipline that treats any unexplained change in engine output as a bug.

Unit & math suites

Futures math, adaptive SL/TP, position sizing, the dollar-risk planner, the trader toolkit, symbol mapping, forex sizing, and market-hours all have dedicated checks.

Adapter proofs

The OANDA adapter is verified against a scripted fake client that asserts the exact request bodies sent to the venue — units, bracket prices, order type — and the close-emulation rules.

Live-engine rig

A mock exchange drives the real engine through entry, stop-fill, partial-take-profit, restart reconciliation, and failed-stop flatten — the full lifecycle, end to end.

Parity harness

The real backtest engine runs a fixed dataset; any change to the decision core must show a zero delta on the crypto path unless the change is intended and measured.

Regression discipline

When the forex market, the new strategy catalog, and the cost-model fixes were added, the crypto engine's backtest output was proven byte-identical to its prior baseline on both spot and futures — new capability, zero disturbance to what already worked.

12

Deployment, operations & scaling

The platform is self-hosted and deploys as a single service. A one-command update script pulls, installs, applies additive schema changes, builds the frontend then the backend, and restarts the process under a supervisor configured to survive crashes without thrashing.

  • Health. A liveness endpoint (is the process up) and a readiness endpoint (can it reach its database) support load balancers and uptime monitors.
  • Resilience. On restart, every engine whose persisted desired state is "running" auto-resumes, so a deploy or reboot never silently halts trading.
  • Observability. Structured logging throughout; uncaught exceptions and unhandled rejections are captured and can be forwarded to an operator's alert webhook.

12.1The single-process model — a deliberate choice

The engine keeps live trading state in one process and must not be clustered: two instances would evaluate the same strategies against the same account and place duplicate orders. This is the correct default for correctness and comfortably serves tens of concurrent users on one machine. It is also the platform's principal scaling boundary — see §13.

13

Honest limitations & roadmap

The platform is offered on the strength of its engineering, not on a track record. Stated plainly, so due diligence holds no surprises:

No verified profitability

The bundled strategies are a competent starting point, not a guaranteed edge. No live-money performance record is claimed. A buyer should expect to validate and tune strategies — the Backtester, Autopsy, and Edge Forensics exist precisely to make that rigorous.

Single-process scaling ceiling

Serving hundreds or thousands of concurrent users would require moving engines to distributed workers with shared state — a scoped, well-understood piece of work, not a rewrite, but real work a SaaS operator would undertake.

Operational & commercial gaps

There is no built-in billing or admin console, and no bundled third-party error-monitoring SDK. Index-CFD stops can slip beyond plan in a fast spike (flagged by the risk audit); majors and gold behave to plan.

None of these are hidden defects — they are the honest edges of a cleanly-built platform, and each is a defined next step rather than an unknown. A prospective owner inherits a coherent, documented, test-defended codebase and a clear map of where to take it.
